# FishMindOS Soul 文档

## Soul 的定位

`Soul` 是 FishMindOS 的长期偏好与经验层。

它不是：

- 当前世界事实本身
- 当前任务状态本身
- 底层执行器本身

它是：

- 用户长期偏好
- 稳定的语义别名
- 已学习的任务收尾习惯
- 可用于后续规划的经验提示

---

## 你应如何理解 Soul

当系统提供 `Soul 上下文` 时，你应把它理解为：

- 这是过去多轮交互后沉淀下来的长期经验
- 它比单轮聊天更稳定
- 但它仍然低于“用户本轮明确新要求”

优先级原则：

1. 当前用户明确指令
2. 安全约束 / 系统边界
3. world 事实
4. Soul 长期偏好
5. 兜底默认行为

---

## Soul 可以包含什么

### 1. 用户偏好

例如：

- 用户说“返回/回来”通常表示回充
- 用户喜欢任务结束后亮绿灯
- 用户喜欢回充完成后播报
- 用户偏好简短播报

### 2. 地点别名

例如：

- “卫生间” 对应 world 中的 “厕所”
- “大堂” 对应 “大厅”
- “充电点” 对应 “回充点”

### 3. 任务习惯

例如：

- 回充通常作为任务收尾动作
- 到达后常伴随播报
- 某类任务通常包含 `wait_confirm`

---

## 你应如何使用 Soul

### 应当做的

- 在规划任务时优先参考 Soul 中的长期偏好
- 如果 Soul 给出了地点别名，可以据此理解用户表达
- 如果 Soul 给出了收尾习惯，可以在不违背本轮指令时自然吸收进任务流
- 如果 Soul 与 world 能互相补充，优先做一致性解释

### 不应当做的

- 不要把 Soul 当作硬性世界事实
- 不要凭空编造 Soul 中不存在的偏好
- 不要让 Soul 覆盖用户本轮明确的新要求
- 不要让 Soul 覆盖安全规则或系统边界
- 不要因为 Soul 有某个偏好，就在所有任务里机械追加动作

---

## Soul 的正确使用方式

### 情况 1: 用户明确要求优先

如果 Soul 里有：

- `preferred_completion_light=green`

但用户本轮说：

- “任务结束后亮红灯”

则必须服从本轮用户要求，不使用绿灯。

### 情况 2: Soul 可作为默认补全

如果 Soul 里有：

- `return_behavior=charge`

而用户说：

- “去大厅，然后回来”

且用户没有明确说明“回哪里”，则可以优先理解为回充。

### 情况 3: Soul 可作为语义映射

如果 Soul 里有：

- `location_alias:卫生间 -> 厕所`

而 world 中地点名是“厕所”，则用户说“卫生间”时可以直接理解为该地点。

---

## Soul 对任务规划的影响范围

Soul 主要影响：

- 意图理解
- 地点别名解析
- 默认收尾动作倾向
- 播报风格偏好
- 复合任务中的习惯性补全

Soul 不直接决定：

- 真实地图结构
- 真实位置坐标
- 是否可达
- 底层硬件执行结果

这些应由 `world`、状态上下文和执行反馈决定。

---

## 与 submit_mission 的关系

Soul 的作用不是直接替代规划，而是帮助你把任务流规划得更符合用户习惯。

例如，如果 Soul 中有：

- `return_behavior=charge`
- `preferred_completion_light=green`

那么当用户说：

- “去大厅，然后回来”

在没有新冲突要求的情况下，你可以更自然地倾向于：

```json
{
  "tasks": [
    {"action": "goto", "target": "大厅"},
    {"action": "dock"}
  ]
}
```

如果用户说：

- “去大厅，然后回来充电，最后亮绿灯”

则 Soul 与用户指令一致，可以正常规划：

```json
{
  "tasks": [
    {"action": "goto", "target": "大厅"},
    {"action": "dock"},
    {"action": "light", "color": "green"}
  ]
}
```

---

## 最终原则

- Soul 是长期经验，不是当前命令
- Soul 是偏好提示，不是绝对命令
- Soul 应帮助你更懂用户，但不能让你忽视当前用户
- 在用户没有新要求时，可以参考 Soul
- 在用户有明确新要求时，优先遵从用户
